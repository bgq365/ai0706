# 反思题

## 这次实现里做对了什么

这次最关键的正确决策，是把“系统 B 的业务闭环”与“系统 A 的上游联调职责”分开处理，而不是把所有内容都塞进一个项目里。这样做以后，系统边界更清晰，提交材料里的“系统 A 地址”和“系统 B 地址”也更容易自证，不会在答辩或验收时陷入“这个页面到底算不算上游系统”的争议。

第二个比较正确的点，是在系统 B 中保留了“实时调用优先、快照回退兜底”的思路。真实业务里，上游系统不稳定是高概率事件，如果一味追求完全实时，系统 B 很容易因为上游抖动而不可用；如果完全不实时，又会失去运单真实性校验的意义。所以这次采用“关键校验实时化、展示可回退”的策略，更接近真实生产设计。

## 这次暴露出的不足

最大的不足，是一开始对“系统 A 是否必须独立部署”判断得不够早，导致前面一度尝试把旧项目 `ai0605` 解释为系统 A。虽然它在业务背景上接近上游 V2，但接口契约并不完全匹配，这种做法用于提交材料会有诚实性风险。后面改成独立新建最小 system A，才把问题真正解决。

第二个不足，是本机网络环境对 `*.vercel.app` 存在访问干扰，影响了部署后的直接浏览器验证。这说明除了代码和接口，我们还需要更重视本地网络、代理、隧道、DNS 优先级这些“非代码因素”，因为它们同样会影响交付质量和验收体验。

## 如果再做一轮，我会怎样优化

如果再做一轮，我会在项目一开始就先固定三件事：

1. 明确系统边界，直接决定是否拆分 system A 和 system B。
2. 先写对接契约，再写业务页面，避免后期补接口。
3. 更早搭建提交材料文档，把“需求理解、假设、反思、模型说明”同步沉淀，而不是最后集中整理。

另外，如果时间再多一些，我会继续补两块：

1. 让 system A 的数据也具备持久化能力，而不只是内存样例。
2. 把 system B 的 Supabase 落库路径彻底打通到全部审批动作，而不是目前以 mock store 为主、Supabase 适配为辅。
