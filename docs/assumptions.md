# 《需求理解与假设说明》

## 核心假设
- V3 与 V2 是两个独立系统，分别部署在独立的 Vercel 项目中。
- V3 使用独立账号/RBAC，而不是直接复用 V2 登录态。
- 若 V2 当前测试数据未区分租户，则暂按单租户 + 仓库范围隔离实现。

## 留白点补全
1. 一级/二级审批阈值：默认以 `1000 元` 为分界，后续落表可配置。
2. 审批超时时长：一级 `2 小时`，二级 `4 小时`。
3. 重提次数上限：默认 `2 次`，超过后自动关闭工单。
4. 物流异常类型映射：
   - 客户拒收：重新派送或退回
   - 超时未签收：重新派送
   - 破损/丢件：赔付客户
5. 角色权限划分：
   - `reporter` 只能发起和提交
   - `level1_approver` 只能一级审批
   - `level2_approver` 只能二级审批
   - `qc_supervisor` 负责快速放行与品控权限
6. V2 数据同步策略：关键动作实时校验，非关键展示使用快照。
7. 品控暂扣超时时长：`90 分钟`，独立于审批超时。
8. 品控规则阈值：数量差异、破损等级、标签异常都应落配置表。
9. 品控主管权限边界：可快速放行但必须留痕，不与普通审批人权限混同。

## 一致性策略
- 工单状态流转与赔付/库存联动最终需要事务或 outbox 模式保证一致性。
- 当前原型只落下接口边界与状态字段，真实数据库阶段补事务实现。
