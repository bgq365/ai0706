# 需求理解与假设说明

## 核心理解

- V3 和 V2 是两个独立系统，分别独立部署、独立数据库。
- V3 不能直连 V2 数据库，运单真实性、详情拉取、SKU 归属校验都必须通过 V2 HTTP API。
- V3 的核心难点不是简单 CRUD，而是审批状态机、品控扫描链路、跨系统一致性和留白规则补全。

## 九项关键假设

### 1. 分级审批金额阈值

- 默认阈值：`1000 元`
- 规则：
  - `< 1000` 走一级审批
  - `>= 1000` 升级二级审批
- 依据：小额异常优先追求处理效率，大额异常需要更高责任级别确认。

### 2. 审批超时时长

- 一级审批超时：`2 小时`
- 二级审批超时：`4 小时`
- 依据：一级审批以快速分流为主，二级审批需要更多判断时间。

### 3. 重提次数上限

- 默认上限：`2 次`
- 第 3 次再被驳回后自动关闭工单
- 依据：避免同一异常在低质量信息下反复流转，占用审批资源。

### 4. 物流异常类型映射

- 客户拒收：重新派送或退回入库
- 超时未签收：重新派送
- 地址错误：修正地址后重新派送
- 丢件/破损：生成赔付记录，赔付方向为 `customer_compensation`
- 依据：按物流常见售后路径设计，保持动作和异常类型一一对应。

### 5. 角色权限划分

- `reporter`：只能发起、编辑、重新提交自己负责的工单
- `level1_approver`：只能处理一级审批
- `level2_approver`：只能处理二级审批
- `qc_supervisor`：可对品控工单执行快速放行，但必须留痕
- `admin`：拥有跨仓和兜底处理权限
- 依据：前后端都要校验，不能只靠前端隐藏按钮。

### 6. V2 数据同步频率与一致性策略

- 关键动作：
  - 发起异常上报
  - 扫描真实性校验
  - SKU 归属校验
  - 以上都优先实时调用 V2
- 非关键展示：
  - 允许使用本地快照回退
- 依据：真实性优先的动作必须实时校验，展示类页面优先保证系统可用性。

### 7. 品控暂扣超时时长

- 默认值：`90 分钟`
- 且必须独立于审批超时
- 依据：品控暂扣代表货物占仓和出库阻塞，业务成本高于单纯审批等待，所以需要更短时限。

### 8. 品控规则触发阈值

- 外观破损：默认直接判定异常
- 标签错误：默认直接判定异常
- 数量差异：默认差异即异常，后续应配置化
- 规则命中需要记录 `ruleId`
- 依据：题目要求规则可配置、可追溯，不能完全写死在代码里。

### 9. 品控主管权限边界

- 可以快速放行品控工单
- 不能越权处理普通物流异常审批
- 不能绕过留痕
- 快速放行必须记录审批动作和意见
- 依据：快放是例外兜底，不应变成通用审批捷径。

## 一致性策略

- 审批通过后触发的赔付记录、库存联动、扫描批次释放，必须能追溯到具体审批记录。
- 当前实现已保留 `approvalRecordId` 级别的追踪字段。
- 如果完全切到真实数据库，建议进一步用事务或 outbox 保证“审批状态更新”和“下游联动记录”同成同败。

## 如果能继续向产品经理追问，我会确认的问题

1. 分级审批阈值是否按仓库、异常类型、金额区间分别配置。
2. 品控规则是否需要按 SKU 类别、供应商、仓库做差异化。
3. 快速放行是否需要二次复核，还是只要求留痕。
4. V2 是否已有稳定对外接口，还是需要在 V2 内新增兼容接口层。
5. 赔付记录是否未来要对接真实财务系统，还是仅需内部留档。
